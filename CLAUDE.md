## Project Overview

AICook is an AI-powered recipe management app (Chinese-language). Users upload recipes, get daily meal recommendations, ask "what can I cook with what's in my fridge", and generate weekly meal plans. The repo is a monorepo with three independent services plus a WeChat mini-program.

## Repository Layout

```
backend/          Go monolith (Kratos v2 + Eino AI framework)
frontend/         React SPA (Vite + Tailwind + Capacitor for mobile)
wx/               WeChat mini-program (native + TypeScript, no Taro/uni-app)
deploy/sql/       Database migrations and seed data
```

## API Serialization Contract (critical ŌĆö easy to get wrong)

1. **JSON is snake_case.** `backend/cmd/backend/main.go` sets `kratosjson.MarshalOptions.UseProtoNames = true`, so all HTTP responses use raw proto field names (NOT protojson's default camelCase). Client types must be snake_case; `pnpm gen:types` is configured accordingly (`snakeToCamel=false, forceLong=string, useDate=string`) ŌĆö keep those options in sync.
2. **IDs are snowflake int64 that exceed JS Number precision (2^53).** Generated by `backend/internal/utils/snowflake.go` (epoch 2025-01-01; values Ōēł1.9e17 by 2026). Clients must hold IDs as strings, use `String(id)` before URLs/bodies, and never `Number(id)` ŌĆö protojson accepts string for int64 fields. Usage rules are documented on `Int64Like` in `wx/miniprogram/types/api.d.ts`.
3. **Meal-plan `days` is a strongly-typed `map<string, MealPlanDaySlots>`** (was `google.protobuf.Struct`, whose numbers are float64 and silently corrupt snowflake IDs ŌĆö changed 2026-06). protojson now auto-converts the int64 `id`/`recipe_id` to strings; don't hand-roll conversions. Outer keys are weekday names `monday..sunday` (not date strings); dishes use `recipe_title` (not `title`). PUT is a full-week overwrite ŌĆö always merge into the existing plan before saving.
4. **Media URLs are signed on read.** Store `MediaAsset.storage_url` (e.g. into `cover_image_url`); the service layer re-signs by host/path on every response (`signRecipeMediaURLs` in `internal/service/helpers.go`). The proto field is `storage_url`, not `url`. Uploads are two-step: PrepareMediaUpload ŌåÆ presigned PUT ŌåÆ CompleteMediaUpload.

## Architecture

### Backend Layering (Kratos pattern)

```
service/   ŌåÆ  HTTP/gRPC handlers (thin, delegates to biz)
biz/       ŌåÆ  Business logic, split by domain (ai/, kitchen/, user/, common/)
data/      ŌåÆ  Repository implementations (PostgreSQL, Redis)
platform/  ŌåÆ  Infrastructure adapters (AI, storage, cache, embeddings, identity)
conf/      ŌåÆ  Configuration loading
server/    ŌåÆ  HTTP/gRPC server wiring and middleware
auth/      ŌåÆ  JWT claims
```

Wire (Google Wire) for compile-time DI. Entry point: `backend/cmd/backend/main.go`, generated code in `wire_gen.go`.

### Auth

JWT middleware applies to all proto routes; the whitelist (no token) is only `AuthService/{Register,Login,WxLogin}` (`internal/server/http.go`). Identity flows via `common.ActorFromContext` (household_id + user_id from claims); the fallback to `DefaultActor` logs a warning and indicates a missing-auth code path ŌĆö never rely on it. Clients also send `X-Household-Id`.

### Streaming Chat ŌĆö the one endpoint outside proto

`POST /chat/send` is a native net/http SSE handler (`internal/server/chat_http.go`), NOT proto-generated, because Kratos handlers are single-response. It bypasses Kratos middleware and does its own JWT check (401 with Kratos-style envelope `{code, reason, message}` on failure). SSE event types (start/answer_delta/reasoning_delta/status_delta/tool_call/recipe_card/agent_delta/approval/done/error) are mirrored in `wx/miniprogram/types/sse-events.d.ts` ŌĆö update both sides together; this is not covered by codegen.

### API Layer

Protocol Buffers in `backend/api/aicook/v1/` (buf-managed; vendored `google/api` protos live under `backend/google/`). Services: AI, Auth, Recipe, Cooking, Kitchen, Knowledge, Media, Voice, Import, Household. Errors use the Kratos envelope `{code, reason, message}`; there is no extra `{code, message, data}` wrapper around responses.

### AI Integration

- **Eino framework** (ByteDance/CloudWeGo) for AI orchestration with ADK and Graph modes
- **AI provider**: Xiaomi MiMo (OpenAI-compatible API)
- **Embeddings**: Doubao multimodal embeddings (2048 dimensions), stored in PostgreSQL via pgvector
- AI runtime code is in `backend/internal/platform/airuntime/`

### WeChat Mini-Program Structure

- `services/` ŌĆö `http.ts` is the single request wrapper (token injection, Kratos error envelope parsing, 401 ŌåÆ clear auth + reLaunch to login, default timeouts). `sse.ts` consumes `/chat/send` via `wx.request({enableChunked: true})` + hand-rolled UTF-8 decoding. `upload.ts` wraps the two-step presigned upload. One `*.api.ts` per backend service.
- `store/` ŌĆö mobx-miniprogram global stores (auth/chat/plan/inventory/household/cooking/ui). `auth.store` emits eventbus events (`HOUSEHOLD_SWITCHED` etc.); pages showing household-scoped data must subscribe to invalidate their caches.
- `types/generated/` ŌĆö authoritative, produced by `pnpm gen:types`; the handwritten `types/api.d.ts` is being migrated away from ŌĆö don't add new proto-mirroring types there.
- UI is hand-rolled BEM-style components (`components/`); tdesign-miniprogram is installed but intentionally unused in pages ŌĆö follow the existing custom-component pattern (e.g. `input-dialog` replaces the deprecated `wx.showModal({editable:true})`).
- Pages cache with a 30s `_lastLoadAt` TTL checked in `onShow`.

### Frontend Structure

Feature-based organization under `frontend/src/features/` (cooking, home, knowledge, plan, profile, recipes, shopping). State management: Zustand with browser persistence. UI: Radix UI + Ant Design X + Tailwind CSS. Mobile: Capacitor for iOS/Android builds. All API calls go through `src/lib/api/client.ts`.

### AI Media Recognition (replaces the old inference-service)

Heavy ML now runs through the MiMo API, not a separate Python service:
- **Speech-to-text**: `internal/platform/asr` posts audio (base64 data-URL, mp3/wav) to MiMo `mimo-v2.5-asr` via `/chat/completions`. Used by `VoiceService.Transcribe`.
- **ImageŌåÆrecipe**: the multimodal `vision_model` ingests image URLs directly (`airuntime` draft path); there is no OCR step.

## Configuration

Backend config: `backend/configs/config.yaml` (or set `AICOOK_CONFIG` env var).

Key services the backend depends on:
- **PostgreSQL** (schema: `aicook`) with pgvector extension
- **Redis** for caching
- **MinIO** for object storage (buckets: `aicook-media`, `aicook-kb`)
- **MiMo API** (Xiaomi, OpenAI-compatible) for chat / vision / ASR

## Database

Migrations in `deploy/sql/`. Key files:
- `base.sql` ŌĆö schema creation
- `seed_demo_recipes.sql` ŌĆö demo recipe data
- `seed_kitchen_tags.sql` ŌĆö kitchen tag presets

The schema uses pgvector for embedding storage and JSONB columns for flexible recipe data.

## Deployment

Kubernetes manifests in `k8s-deployment.yaml` (namespace: `aicook`). Docker multi-stage builds: `golang:1.25` ŌåÆ `alpine`. Backend ports: 8000 (HTTP), 9000 (gRPC).

## Standing Product Decisions

- Knowledge base UI exists only on web (`frontend/src/app/pages/KnowledgeBase.tsx`); the mini-program keeps `services/knowledge.api.ts` as a reserved stub ŌĆö don't build wx UI for it or delete the stub without asking.
- Mini-program dark mode is explicitly deferred (see `wx/README.md`).
- Recipe share codes only clone recipes into the importer's household ŌĆö there is no "join household" semantic.
- Don't display fabricated data in wx pages (match percentages, serving sizes, etc.): bind real API fields or hide the element. The proto `Recipe` has no `servings` field.
